## 1. 游戏基础玩法

### 1.1 游戏概述

这是一款结合了**俄罗斯方块**和**塔防**元素的创新游戏。玩家需要在准备阶段通过放置俄罗斯方块来构建英雄阵容，然后在战斗阶段让英雄自动攻击从顶部涌来的敌人。

### 1.2 核心游戏阶段

#### 阶段一：准备阶段 (Preparation)

**目标**：在限定时间和方块数量内，尽可能多地放置方块并消除行获得分数和能量。

**规则**：
- 时间限制：默认60秒（可通过关卡配置调整）
- 方块限制：默认30个（可通过关卡配置调整）
- 操作方式：
  - ← → 左右移动方块
  - ↑ 旋转方块
  - ↓ 加速下落
  - 空格键 硬降（直接落到底部）
  - P键 暂停

**奖励机制**：
- 消除1行：100分 + 10能量
- 消除2行：300分 + 30能量
- 消除3行：500分 + 50能量
- 消除4行（Tetris）：800分 + 100能量

**特殊机制**：
- 放置的每个方块都会变成英雄
- 方块带有元素属性（火、水、土）
- 消除行后，上方方块会下落

#### 阶段二：战斗阶段 (Battle)

**目标**：利用准备阶段建造的英雄阵容，击败所有波次的敌人。

**规则**：
- 敌人从顶部随机列出现并向下移动
- 英雄自动攻击范围内的敌人
- 敌人到达底部会造成伤害
- 击败所有敌人进入下一波
- 玩家生命值归零时游戏结束

**胜利条件**：击败所有波次的敌人

### 1.3 英雄系统

#### 基础元素英雄

| 元素 | 符号 | 基础伤害 | 攻击范围 | 攻击速度 | 特殊效果 |
|------|------|----------|----------|----------|----------|
| 火 | 🔥 | 10 | 1格 | 1000ms | 无 |
| 水 | 💧 | 8 | 2格 | 1200ms | 减速（50%） |
| 土 | 🌍 | 12 | 1格 | 1500ms | 眩晕（1秒） |

#### 英雄等级系统

- **Lv1**：基础英雄，伤害×1，范围加成0
- **Lv2**：中级英雄，伤害×2，范围加成+1
- **Lv3**：高级英雄，伤害×4，范围加成+2

**升级条件**：3个相邻的同等级、同元素英雄自动合成更高等级。

#### 融合英雄系统

|------|----------|------|------|------|----------|----------|

| 水+水 | 海啸 | 🌊 | 30 | 4格 | 1000ms | 减速 |
| 土+土 | 地震 | ⛰️ | 50 | 2格 | 1200ms | 眩晕 |

| 水+土 | 泥流 | 🟤 | 20 | 3格 | 1100ms | 陷阱（减速70%） |

### 1.4 敌人系统

## 2. 偷懒/作弊玩法分析

### 2.1 严重作弊漏洞

#### 漏洞1：准备阶段方块没有自动下落（最严重）

**问题描述**：
在准备阶段，方块不会像传统俄罗斯方块那样自动下落。玩家可以无限期地保持当前方块在手中，仔细规划布局。

**代码位置**：
- `game.js:185-197` - `updatePreparationPhase()` 方法只更新时间，没有处理方块自动下落

**作弊方式**：
1. 玩家可以按P键暂停游戏，然后有充足时间思考最佳布局
2. 即使不暂停，方块也不会下落，玩家可以慢慢移动和旋转
3. 可以仔细计算如何放置方块以最大化英雄数量和合成机会

**影响**：
- 准备阶段的时间限制变得毫无意义
- 方块数量限制成为唯一的限制因素
- 游戏平衡性被严重破坏

---

#### 漏洞2：游戏状态全局暴露

**问题描述**：
游戏核心对象被暴露到 `window` 全局作用域，玩家可以通过浏览器控制台直接修改游戏状态。

**代码位置**：
- `main.js:14-17`：
```javascript
window.game = game;
window.uiManager = uiManager;
window.canvas = canvas;
window.nextCanvas = nextCanvas;
```

**作弊方式**：
在浏览器控制台（F12）中执行以下命令：

```javascript
// 直接修改分数
game.score = 999999;

// 修改等级
game.level = 5;

// 无限生命值
game.playerHP = 100;
game.maxPlayerHP = 100;

// 跳过准备阶段直接进入战斗
game.skipPreparation();

// 随意切换阶段
game.phase = 'battle';

// 获取更多能量
game.energy = 999;

// 修改已使用方块数
game.preparationPiecesUsed = 0;

// 暂停/继续游戏
game.togglePause();

// 甚至可以直接调用游戏结束
game.gameOver(true); // 直接胜利
```

**影响**：
- 玩家可以完全掌控游戏
- 分数、成就、关卡进度都可以被随意修改
- 游戏失去任何挑战性

---

#### 漏洞3：本地存储数据无保护

**问题描述**：
游戏进度、成就、统计数据都以明文JSON格式存储在 `localStorage` 中，没有任何加密或校验。

**代码位置**：
- `storage.js` - 所有存储函数

**存储的键名**：
- `tetris_tower_high_scores` - 最高分
- `tetris_tower_achievements` - 已解锁成就
- `tetris_tower_level_progress` - 关卡进度
- `tetris_tower_game_stats` - 游戏统计
- `tetris_tower_settings` - 设置

**作弊方式**：
1. 打开浏览器开发者工具（F12）
2. 切换到 Application 标签
3. 找到 Local Storage 中的游戏数据
4. 直接修改JSON值

示例修改：
```javascript
// 解锁所有成就
localStorage.setItem('tetris_tower_achievements', JSON.stringify([
    'first_win', 'tetris', 'merge_hero', 'kill_100', 
    'level_5', 'no_damage', 'merge_master', 'all_levels'
]));

// 解锁所有关卡
localStorage.setItem('tetris_tower_level_progress', JSON.stringify({
    '1': {won: true, score: 10000},
    '2': {won: true, score: 20000},
    '3': {won: true, score: 35000},
    '4': {won: true, score: 50000},
    '5': {won: true, score: 100000}
}));

// 修改统计数据
localStorage.setItem('tetris_tower_game_stats', JSON.stringify({
    totalGames: 100,
    totalScore: 999999,
    totalKills: 1000,
    totalMerges: 50,
    totalTime: 3600,
    levelsCompleted: [1,2,3,4,5],
    perfectDefenses: 10
}));
```

**影响**：
- 玩家可以随意解锁所有关卡和成就
- 统计数据可以被伪造
- 游戏的进度系统完全失效

### 2.2 中等作弊漏洞

#### 漏洞4：暂停功能滥用

**问题描述**：
暂停功能可以在准备阶段使用，让玩家有无限时间思考布局。

**代码位置**：
- `game.js:501-509` - `togglePause()` 方法
- `ui.js:156-159` - 键盘事件处理

**作弊方式**：
1. 在准备阶段按P键暂停
2. 有无限时间思考如何放置方块
3. 想好后继续游戏放置方块
4. 可以重复此过程

**影响**：
- 准备阶段的时间限制失去意义
- 玩家可以完美规划布局

---

#### 漏洞5：硬降刷分

**问题描述**：
硬降（空格键）每下落一行会加2分，玩家可以利用这一点刷分。

**代码位置**：
- `game.js:339-350` - `hardDrop()` 方法

**作弊方式**：
1. 不使用自动下落（本来就没有）
2. 手动将方块移动到底部附近
3. 使用硬降获得额外分数
4. 或者直接反复硬降刷分

**影响**：
- 分数可以被人为抬高
- 游戏分数系统不够严谨

---

#### 漏洞6：英雄布局策略作弊

**问题描述**：
由于方块没有自动下落，玩家可以刻意安排英雄布局以获得最大战斗优势。

**作弊方式**：
1. 不在顶部消除行，而是在底部密集放置英雄
2. 刻意让相同元素相邻以便合成高级英雄
3. 不追求高分，只追求最优战斗布局
4. 利用水元素的范围优势，将水英雄放在中间位置

**影响**：
- 战斗阶段变得过于简单
- 游戏的策略性变成了"如何作弊"

---

### 2.3 轻微漏洞/设计缺陷

#### 漏洞7：缺少反作弊检测

**问题描述**：
游戏没有任何检测异常游戏行为的机制。

**可能的异常行为**：
- 分数增长过快
- 单次游戏分数异常高
- 短时间内完成多个关卡
- 完美防守次数异常

**影响**：
- 无法识别作弊玩家
- 排行榜（如果有的话）会被污染

## 3. 现有问题分析

### 3.1 游戏逻辑问题

#### 问题1：准备阶段缺少方块自动下落机制

**严重程度**：严重

**问题描述**：
如作弊分析中所述，准备阶段的方块不会自动下落，这是最核心的游戏机制缺失。

**代码位置**：
- `game.js:185-197` - `updatePreparationPhase()` 方法

**对比传统俄罗斯方块**：
传统俄罗斯方块会有一个下落计时器，每隔一定时间（根据等级调整）方块自动下落一行。本游戏完全缺失此机制。

**影响**：
- 游戏节奏被破坏
- 时间限制失去意义
- 玩家可以无限思考

---

#### 问题2：成就检查代码与定义不匹配

**严重程度**：中等

**问题描述**：
成就检查代码中引用了不存在的成就ID。

**代码位置**：
- `achievements.js:14-18`：
```javascript
if (!this.isUnlocked('combo_5') && stats.maxCombo >= 5) {
    this.unlock('combo_5');
    newlyUnlocked.push(this.getAchievement('combo_5'));
}

if (!this.isUnlocked('merge_10') && totalStats.totalMerges >= 10) {
    this.unlock('merge_10');
    newlyUnlocked.push(this.getAchievement('merge_10'));
}
```

但在 `config.js:305-314` 中定义的成就ID是：
- `merge_hero`（不是 `merge_10`）
- 没有 `combo_5` 这个成就

**影响**：
- `combo_5` 成就永远不会触发（因为没有定义）
- `merge_10` 成就检查会失败（因为实际ID是 `merge_hero`）
- 成就系统部分功能失效

---

#### 问题3：元素分配算法有偏差

**严重程度**：轻微

**问题描述**：
方块元素的分配算法不是真正随机的，存在偏向性。

**代码位置**：
- `game.js:121-130`：
```javascript
const elements = ['FIRE', 'WATER', 'EARTH'];
const element = elements[((boardValue - 1) % 3)];
```

这里的 `boardValue` 来自：
- `elements.js:151`：`const cellValue = piece.isMerged ? piece.mergedType : (piece.colorIndex + 1);`

而 `colorIndex` 是：
- `elements.js:68`：`const colorIndex = Math.floor(Math.random() * COLORS.length);`

`COLORS.length` 是6，所以 `colorIndex` 范围是0-5，`boardValue` 范围是1-6。

计算 `(boardValue - 1) % 3`：
- boardValue=1 → 0 → FIRE
- boardValue=2 → 1 → WATER
- boardValue=3 → 2 → EARTH
- boardValue=4 → 0 → FIRE
- boardValue=5 → 1 → WATER
- boardValue=6 → 2 → EARTH

**问题**：
虽然数学上是均匀分布，但 `colorIndex` 和 `element` 是分别随机生成的：
- `element` 在 `createRandomPiece()` 中随机选择（3种选择）
- `colorIndex` 也随机选择（6种选择）

这导致元素和颜色之间没有真正的对应关系，可能造成玩家困惑。

---

#### 问题4：英雄合成立刻应用重力

**严重程度**：中等

**问题描述**：
英雄合成后会立即应用重力，导致英雄位置变化，可能影响玩家的布局策略。

**代码位置**：
- `heroes.js:165-167`：
```javascript
if (hasMerge) {
    this.applyGravity();
}
```

**问题**：
1. 合成只在准备阶段结束时执行一次（`game.js:139-140`）
2. 应用重力后，英雄会下落到底部
3. 玩家在准备阶段的布局可能被改变

**影响**：
- 玩家无法预测战斗阶段英雄的实际位置
- 布局策略可能失效

---

#### 问题5：波次完成检测逻辑

**严重程度**：轻微

**问题描述**：
波次完成检测可能存在边界情况。

**代码位置**：
- `enemies.js:258-260`：
```javascript
isWaveComplete() {
    return this.waveKills >= this.waveEnemies && this.enemies.length === 0;
}
```

**潜在问题**：
- 如果有敌人未被击杀但通过了底部（`reached`），它们会从数组中移除
- 但 `waveKills` 不会增加
- 这可能导致 `waveKills < waveEnemies` 但 `enemies.length === 0`
- 此时波次可能无法正确判断完成

**影响**：
- 极端情况下可能导致波次无法推进

---

### 3.2 UI/显示问题

#### 问题6：开始界面按钮可能重复添加

**严重程度**：轻微

**问题描述**：
`main.js` 中的按钮添加逻辑可能与 `index.html` 中已有的按钮冲突。

**代码位置**：
- `main.js:22-36` - `addExtraButtons()` 函数
- `index.html:723-727` - 已有的按钮

**问题分析**：
`index.html` 中已经有：
```html
<div class="button-controls" style="margin-top: 15px;">
    <button class="game-btn secondary" id="btnLevels">关卡选择</button>
    <button class="game-btn secondary" id="btnAchievements">成就</button>
    <button class="game-btn secondary" id="btnStats">统计</button>
</div>
```

而 `main.js:24` 中的检查：
```javascript
if (buttonControls && !document.getElementById('btnLevels')) {
```

检查的是 `document.getElementById('btnLevels')`，但如果按钮已存在，这个条件为 false，应该不会重复添加。

**但问题是**：
- `index.html` 中按钮的父元素是 `.button-controls`
- `main.js` 中查找的是第一个 `.button-controls`
- 然后在其后面插入新的按钮组

这可能导致按钮布局问题。

---

#### 问题7：英雄攻击范围显示不够明显

**严重程度**：轻微

**问题描述**：
英雄的攻击范围用半透明圆圈显示，可能不够清晰。

**代码位置**：
- `heroes.js:315-319`：
```javascript
ctx.strokeStyle = hero.getColor() + '40';  // 只有40%的不透明度
ctx.lineWidth = 2;
ctx.beginPath();
ctx.arc(centerX, centerY, hero.getRange(), 0, Math.PI * 2);
ctx.stroke();
```

**问题**：
- `'40'` 是十六进制的透明度，相当于约25%的不透明度
- 范围圆圈可能太淡，玩家难以看清

#### 问题9：准备阶段UI信息不够直观

**严重程度**：中等

**问题描述**：
准备阶段的时间条和方块数条方向相反，可能造成困惑。

**代码位置**：
- `renderer.js:154-160` - 时间条：

##### 问题3：战斗阶段绘制顺序

**问题代码位置**：`game.js:244-247`


**绘制顺序**：
1. 先绘制英雄（包括范围圈、方块、符号）
2. 后绘制敌人（包括圆形、血条、效果图标）

**问题分析**：

从视觉逻辑看，有两种设计思路：

**思路A：敌人在英雄上方**（当前实现）
- 敌人是"入侵者"，从上方落下
- 敌人应该"覆盖"在静止的英雄之上
- 这样可以清晰看到敌人的移动轨迹

**思路B：英雄在敌人上方**
- 英雄是"防御者"，位置固定
- 敌人"经过"英雄位置时，应该显示英雄在攻击
- 血条和效果图标应该始终可见

**当前实现的潜在问题**：
- 如果敌人正好移动到英雄位置，敌人会完全覆盖英雄
- 玩家可能看不到是谁在攻击
- 英雄的攻击范围圈会被敌人遮挡

**建议**：
这个顺序可能是故意的设计，但需要确认。如果是问题，可以调整绘制顺序或添加半透明效果。

```
第一个效果图标 Y = 圆心Y + size/2 + 5

效果图标 Y = enemyY + 9 + 5 = enemyY + 14

如果 enemyY = 385（接近底部）：
效果图标 Y = 385 + 14 = 399
画布高度 = 400，图标正好在边缘

如果有多个效果：
第二个图标 Y = 399 + 12 = 411，超出画布底部！
```

**边界情况**：
| 敌人位置 | 血条状态 | 效果图标状态 |
|---------|---------|-------------|
| Y < 20 | 部分或全部被截断 | 正常 |
| Y > 380 | 正常 | 可能被截断 |
| BOSS (size=30) | 更容易被截断 | 更容易被截断 |

### 4.4 基于布局变更的响应式设计方案

#### 4.4.1 设计原则

**核心原则**：不单独调整尺寸，而是通过变更布局方式适应不同宽高比。

**布局模式分类**：

```
宽高比 = 宽度 ÷ 高度

┌─────────────────────────────────────────────────────┐
│                    宽高比分类                        │
├─────────────┬─────────────────┬─────────────────────┤
│   竖屏模式   │    均衡模式      │      横屏模式        │
│  (< 0.75:1) │   (0.75 ~ 1.5:1)│     (> 1.5:1)       │
├─────────────┼─────────────────┼─────────────────────┤
│  手机竖屏    │   平板竖屏       │    手机横屏          │
│  (375x812)  │   (768x1024)    │    (812x375)        │
│             │   平板横屏       │    桌面宽屏          │
│             │   (1024x768)    │    (1920x1080)      │
└─────────────┴─────────────────┴─────────────────────┘
```

#### 4.4.2 三种布局模式设计

##### 模式1：竖屏模式（宽高比 < 0.75）

**适用场景**：手机竖屏

**布局结构**：
```
┌─────────────────────────┐
│      游戏标题            │
├─────────────────────────┤
│                         │
│       游戏画布           │
│      (居中显示)          │
│                         │
├─────────────────────────┤
│      信息面板            │
│   (得分、等级、下一方块)  │
├─────────────────────────┤
│                         │
│      控制区域            │
│   方向键 + 功能按钮       │
│                         │
├─────────────────────────┤
│      辅助按钮            │
│   关卡、成就、统计        │
└─────────────────────────┘
```

**关键变更**：
- 信息面板从右侧移到画布下方
- 控制区域保持在底部
- 使用 CSS `transform: scale()` 缩放画布以适应宽度

---

##### 模式2：均衡模式（宽高比 0.75 ~ 1.5）

**适用场景**：平板竖屏、平板横屏、方形屏幕

**布局结构**：
```
┌─────────────────────────────────┐
│            游戏标题               │
├───────────────┬─────────────────┤
│               │   得分面板        │
│   游戏画布     │   等级面板        │
│               │   下一方块        │
│               │                 │
├───────────────┴─────────────────┤
│                                 │
│           控制区域               │
│      方向键 + 功能按钮            │
│                                 │
├─────────────────────────────────┤
│         辅助按钮区域              │
└─────────────────────────────────┘
```

**关键变更**：
- 画布与信息面板水平排列
- 控制区域水平展开
- 根据可用宽度动态调整各区域比例

---

##### 模式3：横屏模式（宽高比 > 1.5）

**适用场景**：手机横屏、桌面宽屏

**布局结构**：
```
┌─────────────────────────────────────────────────────┐
│                    游戏标题                           │
├───────────────┬─────────────────┬───────────────────┤
│               │                 │                   │
│   游戏画布     │    信息面板      │     控制区域        │
│               │                 │                   │
│  (左)         │   (中)          │    (右)           │
│               │  得分            │   方向键           │
│               │  等级            │   功能按钮          │
│               │  下一方块         │   辅助按钮          │
│               │                 │                   │
└───────────────┴─────────────────┴───────────────────┘
```

**关键变更**：
- 三栏水平布局
- 画布可以更高（利用垂直空间）
- 控制区域完全在右侧，不占用底部空间
- 避免UI覆盖游戏区域

#### JavaScript方案：动态计算与缩放

**Canvas 缩放逻辑**：
```javascript
// 在 main.js 或新的 responsive.js 中
class ResponsiveLayout {
    constructor(canvas, nextCanvas) {
        this.canvas = canvas;
        this.nextCanvas = nextCanvas;
        this.baseWidth = 200;   // 原始画布宽度
        this.baseHeight = 400;  // 原始画布高度
        this.currentScale = 1;
        
        this.init();
    }
    
    init() {
        window.addEventListener('resize', () => this.updateLayout());
        this.updateLayout();
    }
    
    getAspectRatio() {
        return window.innerWidth / window.innerHeight;
    }
    
    getLayoutMode() {
        const ratio = this.getAspectRatio();
        if (ratio < 0.75) return 'portrait';   // 竖屏
        if (ratio > 1.5) return 'landscape';   // 横屏
        return 'balanced';                      // 均衡
    }
    
    calculateMaxScale(availableWidth, availableHeight) {
        const scaleX = availableWidth / this.baseWidth;
        const scaleY = availableHeight / this.baseHeight;
        return Math.min(scaleX, scaleY, 2);  // 最大缩放2倍
    }
    
    updateLayout() {
        const mode = this.getLayoutMode();
        const container = document.querySelector('.game-container');
        
        // 移除所有模式类
        container.classList.remove('layout-portrait', 'layout-balanced', 'layout-landscape');
        container.classList.add(`layout-${mode}`);
        
        // 根据模式计算可用空间
        let availableWidth, availableHeight;
        
        switch (mode) {
            case 'portrait':
                // 竖屏：宽度受限
                availableWidth = Math.min(window.innerWidth - 40, 400);
                availableHeight = window.innerHeight * 0.4; // 用40%高度
                break;
                
            case 'landscape':
                // 横屏：高度受限，但可以利用更多宽度
                availableHeight = window.innerHeight - 100;
                availableWidth = this.baseWidth * (availableHeight / this.baseHeight);
                // 但不超过视口的 40%
                availableWidth = Math.min(availableWidth, window.innerWidth * 0.4);
                break;
                
            default: // balanced
                availableWidth = Math.min(window.innerWidth - 250, 400);
                availableHeight = Math.min(window.innerHeight - 300, 500);
        }
        
        this.currentScale = this.calculateMaxScale(availableWidth, availableHeight);
        
        // 应用缩放到画布容器
        const boardWrapper = document.querySelector('.game-board');
        if (boardWrapper) {
            boardWrapper.style.transform = `scale(${this.currentScale})`;
            boardWrapper.style.transformOrigin = 'top left';
            
            // 调整容器大小以适应缩放后的画布
            const scaledWidth = this.baseWidth * this.currentScale;
            const scaledHeight = this.baseHeight * this.currentScale;
            boardWrapper.style.width = `${scaledWidth}px`;
            boardWrapper.style.height = `${scaledHeight}px`;
        }
        
        // 同步缩放 nextCanvas
        const nextWrapper = document.querySelector('.next-piece');
        if (nextWrapper) {
            const nextScale = this.currentScale * 0.8; // 下一方块稍小
            nextWrapper.style.transform = `scale(${nextScale})`;
        }
    }
}
```
